Memory allocator

这个任务是实现每个CPU的空闲列表，并在CPU的空闲列表为空时进行窃取。最终目标是减少锁争用。由输出可以看出，在kmem锁上的争用总数将大大减少

The root cause of lock contention in kalloctest is that kalloc() has a single free list, protected by a single lock. To remove lock contention, you will have to redesign the memory allocator to avoid a single lock and list. The basic idea is to maintain a free list per CPU, each list with its own lock. Allocations and frees on different CPUs can run in parallel, because each CPU will operate on a different list.

Buffer cache

修改块缓存，以便在运行bachetest时，bcache中所有锁的获取循环迭代次数接近于零。理想情况下，块缓存中涉及的所有锁的计数总和应该为零，但如果总和小于500也可以。修改bget和brelse，以便并发查找和释放bcache中不同的块时，不太可能在锁上发生冲突(例如，不必都等待cache.lock)。必须保持每个块最多缓存一个副本的不变式。

Reducing contention in the block cache is more tricky than for kalloc, because bcache buffers are truly shared among processes (and thus CPUs). For kalloc, one could eliminate most contention by giving each CPU its own allocator; that won’t work for the block cache. We suggest you look up block numbers in the cache with a hash table that has a lock per hash bucket

减少块缓存中的争用比kalloc更加棘手，因为bcache缓冲区是真正在进程(以及cpu)之间共享的。对于kalloc，可以通过给每个CPU自己的分配器来消除大多数争用;这对块缓存不起作用。我们建议你在缓存中使用一个哈希表来查找块号，这个哈希表每个哈希bucket都有一个锁
